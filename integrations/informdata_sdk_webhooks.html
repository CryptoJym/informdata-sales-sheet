Quick architecture snapshot

The InformData integration is intentionally asynchronous. Orders queue on InformData’s side, and the authoritative state flows back through webhooks. Our system must accept multi-event bursts (order status, Need Info, call logs, jurisdiction requirement changes) while preserving order-level sequencing. Ingress & Auth

JWT provider issues per-environment tokens. The SDK is bootstrapped with `createConfiguration`, and outbound orders are posted via `OrdersApi.ordersPost`.

Queue & Partition

Every webhook delivery writes to a durable queue (SQS or Pub/Sub) with the payload persisted in cold storage (S3/GCS). Workers partition by order.id to keep events ordered.

State Machine

A state engine tracks status/substatus transitions, Need Info locks, document sequence numbers, and recurring monitoring enrolments. Commands (patches, document uploads, cancels) emit through the same SDK instance.

Observers & SLAs

Verification jobs reconcile webhook events with the source of truth (`ordersIdGet`), raise alerts for stalled orders (>24h without progress), and produce customer-facing transcripts.

Environments & authentication

InformData issues JWTs per customer and environment. Tokens are short-lived (typically 15 minutes) and must be refreshed via your credential vault or the InformData OAuth partner portal. All endpoints live under HTTPS.

Environment Base URL Notes
Sandbox https://sandbox.api.sjvassoc.com/clients/ Rate limited to ~5k requests / 5 min per IP. Use for replay demos and regression tests.
Production https://api.sjvassoc.com/clients/ All customer traffic. Keep webhook endpoints on dedicated subdomains with WAF/allowlists.
Prerequisites
  • Installed SDK tarball (`typescript-sdk-prod-v0.9.2.tar`) or published npm package.
  • Token broker (AWS Secrets Manager, Doppler, or Vault) to surface fresh JWTs.
  • Queue + storage layer (SQS/SNS + DynamoDB, Pub/Sub + BigQuery, etc.).
Security callouts
  • Webhook origin is not signed — enforce IP allowlists or an additional shared-secret header.
  • Rotate JWT fetch secrets quarterly and alert on 401/403 bursts.

Bootstrap the TypeScript SDK

Wrap SDK creation in a module that injects the current token. The snippet below is the production starter we deploy in both sandbox and production.

import {
  createConfiguration,
  configureAuthMethods,
  OrdersApi,
  WebhooksApi,
  MonitoringApi,
} from '@informdata/connect-client';

export interface InformDataClients {
  orders: OrdersApi;
  webhooks: WebhooksApi;
  monitoring: MonitoringApi;
}

export function getInformDataClients(baseUrl: string, tokenProvider: () => Promise<string>): InformDataClients {
  const configuration = createConfiguration({
    baseServer: { url: baseUrl, description: 'InformData Connect' },
    authMethods: configureAuthMethods({
      'JWT-Authentication': { tokenProvider },
    }),
  });

  return {
    orders: new OrdersApi(configuration),
    webhooks: new WebhooksApi(configuration),
    monitoring: new MonitoringApi(configuration),
  };
}

Add a circuit breaker around every SDK call:

  • Retry 5xx and 429 with capped exponential backoff (1s → 2s → 4s → 8s, max five attempts).
  • Surface 400 validation errors directly in the queue dead-letter topic for triage.
  • Wrap ordersIdGet with caching to avoid hammering the API when reconciling documents.

Order submission & multi-order flow

An order can change states multiple times and generate additional documents long after submission. Persist every state change so downstream systems (billing, customer notifications) stay in sync.

Order POST contract

Use the JSON templates under /informdata-api-docs/Requests as the authoritative schema. They already include jurisdiction, product, and subject permutations.

const { orders } = getInformDataClients(SANDBOX_URL, fetchToken);
const request = buildCriminalOrder(payload); // map UI/request form -> SDK

const response = await retry(async () => orders.ordersPost({
  orderRequest: request,
}));

await orderRepository.recordSubmission({
  id: response.order?.id!,
  clientOrderId: request.order?.clientOrderId!,
  submittedAt: new Date(),
});
Multi-order orchestration
  1. When webhooks arrive, enqueue them with a composite key (`order.id` + `eventType`).
  2. FIFO worker dequeues events for a single order, normalises timestamps, and updates the state chart.
  3. Downstream processors (billing, customer email, dashboards) subscribe to the enriched state stream.

Webhook catalogue & payload anatomy

InformData currently emits four core webhook event types (plus sandbox-only diagnostics). Event bodies mirror ordersIdGet responses.

Event When it fires Primary fields Multi-order implications
ORDER_STATUS_CHANGE Status/substatus transitions (e.g., PENDING/IN_PROGRESS → COMPLETE/RECORD). order.status, order.substatus, order.activity[], order.cases[]. Advance the fulfillment state machine and notify customers when completed.
NEED_INFO InformData pauses the order pending clarification, documents, or authorisation. order.activity (reason), order.documents[] (requested forms). Suspend automation for the order. Create a CSR task and resume only after patching.
CALL_LOG_ADDED A manual note or researcher log is attached to the order. order.activity[*].note, addedDate, optional userName. Fetch latest order to capture attachments referenced in the log. No state change.
JX_REQUIREMENT_CHANGE Jurisdiction requirements (fees, forms) changed mid-search. order.jurisdiction, order.pricing, order.activity. Evaluate whether existing orders need manual intervention or new pricing approval.
InformData currently retries webhooks with exponential backoff for up to 24 hours. If your endpoint returns a non-2xx status, expect duplicates. Always dedupe using order.id and event metadata.

Scaling the multi-order queue

When dozens of webhooks arrive in parallel, we partition by order and sequence by activity timestamp to maintain state consistency.

Queue design checklist
  • Ingress topic: SNS/SQS FIFO or Pub/Sub with ordering key.
  • Partition key: order.id.
  • Idempotency: Hash of order.id + eventType + documents[].sequenceNumber.
  • Dead-letter: 10 failed attempts → send to DLQ with payload snapshot.
  • Storage: archive every payload in S3/GCS for 30 days.
Sequencing algorithm
  1. Sort order.activity by addedDate (ISO string) ascending.
  2. Within identical timestamps, prefer arrival order.
  3. When documents have sequenceNumber, ensure they are appended in order and deduped.
  4. Emit deterministic change events (OrderProgressed, NeedInfoRaised, DocumentReceived).
for await (const event of orderQueue.consume({ groupId: payload.order.id })) {
  const snapshot = await orderRepository.getOrCreate(payload.order.id);
  const applied = applyWebhook(snapshot, payload);

  await orderRepository.upsert(applied);
  await projections.emitOrderTimeline(applied);
}

Resolving Need Info, documents, and recurring monitors

Follow the flows below to unblock orders and manage long-running monitoring products.

Need Info resolution 
await orders.ordersIdPatch({
  id: orderId,
  orderPatchRequest: {
    note: 'Uploaded driver authorisation',
    supplementalDocuments: [
      {
        documentName: 'DriverAuthorisation.pdf',
        documentType: 'LETTER_OF_AUTHORIZATION',
        contentBase64: await encodeFile('loa.pdf'),
      },
    ],
  },
});

After the patch, expect an ORDER_STATUS_CHANGE webhook confirming the order resumed.

Document uploads 
await orders.ordersIdDocumentsPost({
  id: orderId,
  documentUploadRequest: {
    sequenceNumber: 2,
    documentName: 'Disposition.pdf',
    documentType: 'COURT_RECORD',
    contentBase64: await encodeFile('Disposition.pdf'),
  },
});

Uploads trigger a subsequent webhook that references the sequenceNumber. Use it to detect duplicates.

Recurring monitoring (MedEx / Booking / Risk Monitoring)
  • Use orders.ordersIdRecurringDelete to cancel an enrolment.
  • Webhook payloads include order.monitoringDetails with the next check date.
  • Persist the schedule to align billing and SLA commitments.

Verification & monitoring

Use the following checks during rollout and ongoing operations.

Smoke tests
  • Run pnpm test:replay inside the webhook replay demo to ensure state transitions apply cleanly.
  • Submit a sandbox order daily and assert end-to-end completion within 6 hours.
Data integrity
  • Nightly reconcile ordersIdGet against stored state to catch missed webhooks.
  • Alert if an order remains in PENDING/IN_PROGRESS > 72 hours or NEED_INFO > 24 hours.
Security
  • Log every failed webhook auth (unexpected IP or missing shared secret) and alert.
  • Rotate JWT credentials whenever InformData rotates signing keys.

Error handling cheat sheet

Status / webhook symptom Meaning Playbook action
400 from SDK Payload failed InformData validation. Stop retries, capture the validation message, surface to intake UI for correction.
401 / 403 Token expired or IP not on allowlist. Refresh JWT, confirm environment, audit IP restrictions.
409 Order already locked or processed. Wait for the next webhook; do not retry immediately.
5xx / 429 Temporary InformData outage or throttling. Retry with exponential backoff, widen jitter, alert if >15 minutes of consecutive failures.
Webhook duplicates Endpoint returned non-2xx or delivery retried. Rely on idempotency key, log duplicates, instrument delivered vs. applied metrics.
